Section 1 · Part 1 · Architecture

Role & Project Goal Locked

1.1 AI Role Definition

When executing against this directive, the AI assumes the role of Senior Unity Developer and C# Architect. Every response must be implementation-ready — not conceptual, not pseudocode. Real scripts, real Inspector values, real tuning guidance.

1.2 Project Goal

Develop a 2D top-down "Monster Tamer" RPG with a seamless, continuous open world. No loading screens between areas. The player captures, raises, and evolves creatures that follow them in the overworld. The world is large enough to require streaming or chunking to stay performant.

1.3 Style References

AxisReferenceWhat We Take
Visual StyleStardew ValleyPixel-art top-down perspective. Warm palette, readable silhouettes, 16×16 or 32×32 tile base. Layered depth (player walks behind trees). Cozy but not childish.
Creature MechanicsDigimonBranching evolution web (not linear chains). Devolution is always available. Stats matter. Companion creatures follow the player in the overworld. Bond/friendship as a real progression axis.
World FeelStardew × PokémonSeamless open world with distinct biomes. No hard scene transitions. NPC-populated towns. Day/night cycle (stretch goal). The world exists whether the player is there or not.
Bridge to GDD: This document does not replace the Game Design Document (currently v0.7.9). The GDD defines what the game is — its systems, types, creatures, and progression. This Technical Directive defines how to build it in Unity. Every implementation choice here must serve the design decisions locked in the GDD.

Technical Architecture Locked

2.1 Engine & Rendering

ConstraintDecisionRationale
EngineUnity Latest LTSStability over bleeding-edge. LTS versions receive two years of support. Check Unity Hub → Installs → LTS for the current version at time of development.
Render Pipeline2D URP (Universal Render Pipeline)Built-in 2D lighting, shadow casters, sprite sorting support. Required for visual effects (torchlight, day/night). Do not use HDRP or Built-in RP.

2.2 Data Model

SystemStorageWhy
Monster DataScriptableObjectsEach species is an asset. Designers can edit in Inspector without touching code. Supports hot-reload in Editor.
Evolution TreesScriptableObjects (graph)Each node references a MonsterData asset plus conditions. The web structure from the GDD maps directly to a graph of SO references.
Move Sets / SkillsScriptableObjectsSkills reference attributes and types from the GDD's 18-tree taxonomy. Each skill is an individual asset.
Save DataJSON serializationRuntime state (player position, monster stats, bond levels, inventory) serializes to JSON. ScriptableObjects are templates, not save state.
Rule: ScriptableObjects are read-only at runtime. Never write to a ScriptableObject during play — it persists in the Editor and will corrupt your template data. Runtime state goes into plain C# classes that serialize to JSON.

2.3 World Management

The open world is divided into chunks — rectangular groups of tiles and objects stored as child GameObjects of a parent "Chunk" object. Only chunks near the player are active. Distant chunks are disabled (SetActive(false)), not destroyed. This avoids instantiation cost and garbage collection spikes.

Two approaches, choose based on world size:

ApproachWhen to UseHow It Works
GameObject ChunksSmall-to-medium worlds (under ~200 chunks)All chunks exist in one scene. A manager script toggles SetActive based on player distance. Simpler, faster to iterate.
Additive Scene StreamingLarge worlds (200+ chunks, multiple biomes)Each chunk is its own scene. Loaded/unloaded additively via SceneManager.LoadSceneAsync. Higher memory ceiling, more complex to author.
Start with GameObject Chunks. Migrate to Additive Scenes only if profiling shows memory pressure. Premature scene-streaming adds authoring complexity for no player-visible benefit.

2.4 Pathfinding

OptionRecommendationNotes
A* Pathfinding Project (Lite)Primary choiceFree version covers 2D grid graphs. Supports dynamic obstacles, graph updates, seeker components. Well-documented. Import via Asset Store.
Unity NavMesh (2D surface)FallbackRequires NavMeshSurface2D bake. Less intuitive for tile-based worlds. Use only if A* is unavailable or if the project later needs 3D hybrid.

2.5 Performance Constraints

RuleDetail
No heavy logic in Update()Frame-dependent checks (input polling, animation state) are fine. Distance checks, AI decisions, and pathfinding recalculations must use timers, coroutines, or event callbacks.
Event-driven over pollingUse C# event/Action delegates, Unity Events, or a lightweight message bus. Example: monster doesn't poll "am I close to player?" — instead, it subscribes to a proximity trigger event.
Object pooling for spawnsMonsters, projectiles, and VFX that appear/disappear frequently must use a pool. Never Instantiate/Destroy in a loop during gameplay.
Profile before optimizingUse Unity Profiler (Window → Analysis → Profiler). Optimize the spike you can measure, not the one you imagine.

Section 5 · Part 2 · Phase 1

Player Controller Active

5.1 Requirements

  • 8-directional movement (cardinal + diagonal) using Rigidbody2D physics
  • Input via Unity's new Input System (not legacy Input.GetAxis)
  • Diagonal movement must be normalized (no faster diagonal speed)
  • Smooth acceleration and deceleration — not instant start/stop
  • Animation state drives walk/idle sprite based on facing direction
  • Rigidbody2D set to Dynamic with gravity scale 0 (top-down, no falling)

5.2 C# Script — PlayerController.cs

C# PlayerController.cs
using UnityEngine;
using UnityEngine.InputSystem;

/// <summary>
/// Top-down 8-directional player movement using Rigidbody2D.
/// Reads input in Update(), applies physics in FixedUpdate().
/// </summary>
[RequireComponent(typeof(Rigidbody2D))]
public class PlayerController : MonoBehaviour
{
    [Header("Movement")]
    [Tooltip("Max speed in units per second")]
    [SerializeField] private float _moveSpeed = 5f;

    [Tooltip("How quickly the player reaches max speed (higher = snappier)")]
    [SerializeField] private float _acceleration = 50f;

    [Tooltip("How quickly the player stops (higher = less slide)")]
    [SerializeField] private float _deceleration = 40f;

    [Header("Animation")]
    [Tooltip("Reference to the Animator component")]
    [SerializeField] private Animator _animator;

    // ── Cached References ──
    private Rigidbody2D _rb;
    private Vector2 _inputVector;
    private Vector2 _currentVelocity;
    private Vector2 _lastFacingDirection = Vector2.down;

    // ── Animator Hash IDs (avoid string lookups every frame) ──
    private static readonly int AnimMoveX = Animator.StringToHash("MoveX");
    private static readonly int AnimMoveY = Animator.StringToHash("MoveY");
    private static readonly int AnimIsMoving = Animator.StringToHash("IsMoving");

    private void Awake()
    {
        _rb = GetComponent<Rigidbody2D>();
        _rb.gravityScale = 0f;
        _rb.freezeRotation = true;
    }

    // ── Called by the New Input System's PlayerInput component ──
    public void OnMove(InputAction.CallbackContext context)
    {
        _inputVector = context.ReadValue<Vector2>();
    }

    private void Update()
    {
        // Update animator with current facing direction
        bool isMoving = _inputVector.sqrMagnitude > 0.01f;

        if (isMoving)
        {
            _lastFacingDirection = _inputVector.normalized;
        }

        if (_animator != null)
        {
            _animator.SetFloat(AnimMoveX, _lastFacingDirection.x);
            _animator.SetFloat(AnimMoveY, _lastFacingDirection.y);
            _animator.SetBool(AnimIsMoving, isMoving);
        }
    }

    private void FixedUpdate()
    {
        Vector2 targetVelocity = _inputVector.normalized * _moveSpeed;

        // Accelerate toward target, decelerate when no input
        float rate = targetVelocity.sqrMagnitude > 0.01f
            ? _acceleration : _deceleration;

        _currentVelocity = Vector2.MoveTowards(
            _currentVelocity, targetVelocity,
            rate * Time.fixedDeltaTime
        );

        _rb.linearVelocity = _currentVelocity;
    }
}

Section 6 · Part 2 · Phase 1

Sprite Sorting & Depth Logic Active

6.1 The Problem

In a top-down 2D game, the player must walk behind a tree when above it and in front of the tree when below it. Without sorting logic, sprites overlap randomly or based on creation order.

6.2 Solution: Y-Position Sorting

Unity's 2D renderer can sort sprites by their Y position. Objects higher on screen (larger Y) render behind objects lower on screen. Two ways to set this up:

MethodSetupTrade-off
Transparency Sort Axis (global)Edit → Project Settings → Graphics → Transparency Sort Mode: Custom Axis → Y=1, X=0, Z=0Simplest. Works for all sprites on the same sorting layer. May need pivot point adjustment on tall sprites.
Runtime sortingOrder scriptAttach a script that sets SpriteRenderer.sortingOrder = -(int)(transform.position.y * 100)More control per-object. Small performance cost. Use for objects that need special sorting (multi-tile buildings).
Use the global Transparency Sort Axis first. Only add per-object scripts for edge cases (tall buildings, multi-sprite NPCs).

6.3 Sorting Layers

Create these Sorting Layers in order (rendered bottom-to-top):

OrderLayer NameContents
0GroundGrass, dirt, water, paths — always behind everything
1GroundDecorFlowers, puddles, shadows, cracks — on top of ground
2CharactersPlayer, monsters, NPCs — sorted by Y within this layer
3ForegroundTree canopies, roof overhangs — always in front of characters
4UI_WorldFloating health bars, name tags, interaction prompts

6.4 Inspector Setup

Sorting Layer Configuration
Project Settings → Tags & LayersAdd Sorting Layers in order: Ground, GroundDecor, Characters, Foreground, UI_World
Project Settings → GraphicsTransparency Sort Mode → Custom Axis. Transparency Sort Axis → (0, 1, 0)
Tree sprite (trunk)Sorting Layer: Characters. Pivot point at base of trunk (bottom-center). Y-sorting handles the rest.
Tree sprite (canopy)If the tree has a separate canopy overlay, put it on Foreground layer so the player always walks under it.
Player spriteSorting Layer: Characters. Pivot at feet.

6.5 Pivot Point Rule

Critical: Every sprite that participates in Y-sorting must have its pivot point at its feet / base. A tree's pivot is at the bottom of the trunk. The player's pivot is at their feet. If the pivot is centered, a tall tree will sort against its middle, causing visual popping. Set pivots in the Sprite Editor (Sprite Editor → set Pivot to Bottom).

Section 9 · Part 4 · Phase 3

Companion State Machine Pending Phase 3

9.1 States

StateCondition to EnterBehavior
IdlePlayer is stationary and within follow radiusMonster stops moving. Plays idle animation. Faces player.
FollowPlayer moves beyond personal space bufferMonster pathfinds toward player's position. Maintains a buffer distance — doesn't stack on top of player.
Catch UpPlayer moves beyond a "leash" distanceMonster moves at increased speed (1.5× base) to close the gap. Transitions back to Follow once within follow radius.
WanderPlayer has been idle for 5+ secondsMonster picks a random point within a small radius and walks to it. Returns to Idle after reaching the point. Creates the illusion of a living creature exploring nearby.
TeleportDistance exceeds maximum leash (scene transitions, player sprints far)Monster teleports to a point near the player with a brief VFX. Safety net — prevents permanent separation.

9.2 Key Concepts

Personal Space Buffer: The companion never gets closer than a minimum distance to the player. This prevents the "stepping on your feet" problem. The buffer should be roughly 1–1.5 tiles.

Smoothed Arrival: When the companion reaches its target position, it doesn't snap-stop. It decelerates over a short distance. This is the single most important "feels alive" detail.

Bond-Driven Behavior (GDD §6): As bond level increases, the companion's follow distance tightens, wander radius increases (more confident), and it responds faster to player movement. These are tuning values, not state changes.

Section 10 · Part 4 · Phase 3

Pathfinding & Obstacle Avoidance Pending Phase 3

10.1 A* Pathfinding Setup

Import A* Pathfinding Project (Lite) from the Unity Asset Store. The free version supports grid graphs, which is exactly what a tile-based world needs.

A* Pathfinding Configuration
Create: A* GameObjectAdd an empty GameObject named A*. Add the Pathfinder component.
Graph TypeAdd a Grid Graph. Set width/depth to cover your playable area. Node size: 0.5 (half a tile for finer navigation).
Collision TestingSet collision mask to your "Obstacles" physics layer. Diameter: match node size. Height testing off (2D).
Monster PrefabAdd Seeker component + AIPath component (or AILerp for smoother movement). Set max speed to match follow speed tuning.
Obstacle LayerAll trees, fences, buildings, and water tiles must be on an "Obstacles" physics layer with colliders. The grid graph scans these on startup.

10.2 Avoiding "Stuck on Corners" Problem

The most common companion AI bug: the monster gets stuck on the corner of an obstacle because the collider edge catches it. Three mitigations:

  • Use CircleCollider2D on the monster (not BoxCollider2D). Circles slide along edges naturally.
  • Erosion on the grid graph: Set erosion iterations to 1 in the A* graph settings. This makes nodes adjacent to walls unwalkable, keeping the path away from edges.
  • Obstacle padding: Set AIPath.pickNextWaypointDist high enough (0.5–1.0) that the monster doesn't try to hug corners.

10.3 Path Recalculation Frequency

Performance rule: Do not recalculate the path every frame. Use AIPath.repathRate = 0.5f (recalculate every 0.5 seconds). The visual smoothing of AIPath hides the gap between recalculations. If the player hasn't moved, skip the recalculation entirely — subscribe to a "player moved" event rather than polling.

Section 11 · Part 5 · Phase 4

World Chunk Manager Pending Phase 4

11.1 Concept

The world is divided into a grid of chunks. Each chunk is a parent GameObject containing all tiles, objects, NPCs, and creatures for that area. The WorldChunkManager script tracks the player's position and enables/disables chunks based on distance.

11.2 C# Script — WorldChunkManager.cs

C# WorldChunkManager.cs
using UnityEngine;
using System.Collections.Generic;

/// <summary>
/// Enables/disables world chunks based on player proximity.
/// Does NOT use Update() — recalculates on a timer.
/// </summary>
public class WorldChunkManager : MonoBehaviour
{
    [Header("References")]
    [Tooltip("The player's Transform")]
    [SerializeField] private Transform _playerTransform;

    [Header("Chunk Settings")]
    [Tooltip("Distance in units to keep chunks active")]
    [SerializeField] private float _activateRadius = 30f;

    [Tooltip("Distance beyond which chunks are disabled")]
    [SerializeField] private float _deactivateRadius = 40f;

    [Tooltip("Seconds between chunk visibility checks")]
    [SerializeField] private float _checkInterval = 0.5f;

    [Header("Chunks")]
    [Tooltip("Parent objects for each chunk — auto-detected if empty")]
    [SerializeField] private List<GameObject> _chunks = new();

    private float _timer;

    private void Start()
    {
        // Auto-detect chunks if list is empty
        if (_chunks.Count == 0)
        {
            foreach (Transform child in transform)
                _chunks.Add(child.gameObject);
        }

        // Initial evaluation
        EvaluateChunks();
    }

    private void Update()
    {
        // Timer-based, not every frame
        _timer += Time.deltaTime;
        if (_timer < _checkInterval) return;
        _timer = 0f;

        EvaluateChunks();
    }

    private void EvaluateChunks()
    {
        Vector2 playerPos = _playerTransform.position;

        for (int i = 0; i < _chunks.Count; i++)
        {
            float dist = Vector2.Distance(
                playerPos, _chunks[i].transform.position
            );

            bool isActive = _chunks[i].activeSelf;

            // Hysteresis: different thresholds for on/off
            // prevents flickering at boundary
            if (!isActive && dist < _activateRadius)
                _chunks[i].SetActive(true);
            else if (isActive && dist > _deactivateRadius)
                _chunks[i].SetActive(false);
        }
    }
}

11.3 Variable Tuning

Activate Radius
30
units
Range: 20–50. Must be larger than the camera viewport. At 320×180 reference resolution, the viewport is ~20 units wide. 30 gives a buffer of ~5 units off-screen on each side.
Deactivate Radius
40
units
Range: 30–60. Must be larger than activate radius (hysteresis). The gap prevents chunks from flickering on/off when the player is near a boundary.
Check Interval
0.5
seconds
Range: 0.2–1.0. How often the manager checks chunk distances. At 0.5, it runs twice per second — plenty fast for walking speed. Lower if the player can move very fast (sprint, teleport).

12.1 The Update() Rule

Update() runs every frame — 60+ times per second. Anything inside it must be cheap. Here's what's allowed and what isn't:

Allowed in Update()Move to Timer / Event / Coroutine
Input polling (InputAction.ReadValue)Distance checks between objects
Animation parameter updatesPathfinding recalculations
Simple boolean state checksAI decision-making
UI element updates (health bar fill)Chunk visibility evaluation
Camera follow (smooth damp)Save/load operations

12.2 Event-Driven Pattern Examples

Instead of polling, subscribe to events. This is the single biggest performance and architecture win.

C# Event-Driven Pattern — Player Health Example
// ── BAD: Polling in Update ──
void Update() {
    healthBar.fillAmount = player.CurrentHP / player.MaxHP; // runs 60×/sec for no reason
}

// ── GOOD: Event-driven ──
void OnEnable() {
    player.OnHealthChanged += UpdateHealthBar; // subscribe once
}

void OnDisable() {
    player.OnHealthChanged -= UpdateHealthBar; // always unsubscribe
}

void UpdateHealthBar(float current, float max) {
    healthBar.fillAmount = current / max; // runs only when HP changes
}

12.3 Timer Pattern

C# Timer Pattern — Throttled Checks
// For things that need periodic checks but not every frame
private float _aiTimer;
[SerializeField] private float _aiTickRate = 0.25f;

void Update()
{
    _aiTimer += Time.deltaTime;
    if (_aiTimer < _aiTickRate) return;
    _aiTimer = 0f;

    // Expensive logic runs 4×/sec instead of 60×/sec
    EvaluateAIState();
}

12.4 Object Pooling Rule

Any object that is frequently created and destroyed must use a pool. This includes:

  • Wild monster spawns/despawns in chunks
  • Projectile and VFX particles during battle
  • Floating damage numbers
  • Interaction prompt UI elements
Unity 2021+ includes UnityEngine.Pool.ObjectPool<T> as a built-in generic pool. Use it instead of writing your own. For GameObjects specifically, consider a simple wrapper that calls SetActive(true/false) instead of Instantiate/Destroy.
